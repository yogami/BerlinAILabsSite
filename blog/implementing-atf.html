Implementing the Agentic Trust Framework: A Practical Guide

Yami Gopal Berlin AI Labs
February 9, 2026 10 min read

The Cloud Security Alliance released the Agentic Trust Framework (ATF) as a zero-trust security model for AI agents. It's an important spec. But it shipped as documentation only ‚Äî no running code, no reference implementation, no way to test it against real agents. We decided to build one. This is how we did it, what we learned, and how you can validate it yourself.

The Problem: AI Agents Have No Zero Trust

Traditional zero trust (NIST SP 800-207) was designed for human users accessing corporate resources. It answers questions like: "Is this person authorized to access this database?" and "Is this session still valid?"

AI agents break every assumption in that model. They don't have sessions ‚Äî they have continuous autonomous loops. They don't access one resource ‚Äî they chain 30 API calls in sequence. They don't have a fixed intent ‚Äî they adapt their actions based on intermediate results. And when they go wrong, they don't just show an error page ‚Äî they can exfiltrate data, inject prompts into other agents, and make irreversible real-world decisions.

The ATF addresses this by defining 5 trust elements that every agent must satisfy:

Who is this agent? Who owns it? What is its declared purpose?

üëÅÔ∏è Element 2 ‚Äî Behavior

Is the agent doing what it claims? Can we prove it cryptographically?

üõ°Ô∏è Element 3 ‚Äî Data Governance

What data goes in? What comes out? Is PII protected?

üìä Element 4 ‚Äî Segmentation

Where can this agent go? What can it access? Who can it talk to?

‚öîÔ∏è Element 5 ‚Äî Incident Response

What happens when the agent goes rogue? How fast can we contain it?

The spec is well-structured. But turning it into running software meant solving problems the spec deliberately leaves open. Here's what we built.

Our reference implementation isn't a monolith ‚Äî it's a portfolio of 12 independently deployed services, each covering part of the ATF. This mirrors how enterprises actually work: different teams own different trust functions.

What the Spec Doesn't Cover (and We Built)

The ATF spec defines what trust elements exist. It doesn't specify how to implement them at runtime. These are the three biggest gaps we filled:

1. Maturity Model Runtime

The spec describes agent maturity levels (Intern, Junior, Senior, Lead, Director) but doesn't define how an agent earns promotion. We built a runtime engine with 5 concrete Promotion Gates:

// Promotion Gate evaluation const gates = [ { name: 'Performance', check: accuracy > 0.9 && availability > 0.99 }, { name: 'Security', check: pentestGrade <= 'B' }, { name: 'Business Value', check: roi > 0 && ownerApproved }, { name: 'Incident Record',check: criticalIncidents === 0 }, { name: 'Governance', check: securityTeam && riskCommittee }, ]; // All gates must pass for promotion

Each gate is independently verifiable. The performance gate checks accuracy and availability over a configurable window. The security gate needs a pentest score of B or higher from Agent Pentest. The business value gate requires a human to sign off. This gives orgs the scaffolding to evaluate agents their own way, with their own thresholds.

2. Segmentation & Policy Engine

The spec says agents should be segmented by role and permissions. It doesn't say how. We implemented a policy-as-code engine that evaluates access requests in real time:

// Access policy enforcement const result = segmentationService.evaluateAccess({ agentId: 'supply-chain-optimizer-v3', resource: 'procurement-db/write', maturityLevel: 'intern', requestsInWindow: 42, }); // ‚Üí { allowed: false, reason: 'Maturity level insufficient (intern < junior)' }

The engine uses the agent's maturity level, current rate limit status, and resource classification to make access decisions. Denied requests fall back to the next lowest permission level automatically ‚Äî an intern denied write access still gets read access.

3. Incident Response Circuit Breaker

When an agent starts behaving anomalously, you need to contain it before the damage spreads. We built a circuit breaker with three states (CLOSED ‚Üí OPEN ‚Üí HALF_OPEN) that automatically isolates agents exceeding a failure threshold, with configurable cool-down periods and gradual re-admission.

How to Validate It

We don't ask you to trust us. We ask you to run the tests:

git clone https://github.com/yogami/atf-reference-implementation.git cd atf-reference-implementation npm install npm test # ‚Üí 25/25 contract validation tests passing

The contract tests verify: all 5 ATF elements have at least one implementation, no duplicate repos, all services have valid URLs, required capabilities exist per element, every non-infrastructure service has a documented ATF mapping, all promotion gates are represented, and deployed services use HTTPS.
